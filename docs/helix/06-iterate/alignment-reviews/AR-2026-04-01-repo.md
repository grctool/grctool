## Intent Summary

- **Vision**: GRCTool is intended to evolve from a Tugboat-focused CLI into a broader local-first system of record with multi-provider sync, durable IDs, and audit-ready workflows.
- **Requirements**: The PRD and frame stack expect clear command UX, browser auth, local JSON storage, strong evidence workflows, and expanding provider abstractions without breaking Tugboat-driven operations.
- **Features / Stories**: FEAT-005 through FEAT-010 describe the currently shipped core surface. FEAT-001 through FEAT-004 describe the provider-expansion roadmap and lifecycle orchestration that should sit on top of the current core.
- **Architecture / ADRs**: ADR-006, ADR-010, and ADR-011 require extensible provider boundaries, canonical local IDs, and system-of-record semantics governed by the planning stack rather than inferred from implementation.
- **Technical Design**: The current design stack expects provider interfaces, registry-driven orchestration, conflict handling, lifecycle state machines, and testable sync behavior.
- **Test Plans**: The testing strategy requires a 4-tier layout, deterministic no-auth feedback, VCR-backed external integrations, and tests that remain aligned with the canonical feature contracts.
- **Implementation Plans**: Development practices require minimal, in-place refactoring with tests and docs updated alongside behavior changes.
