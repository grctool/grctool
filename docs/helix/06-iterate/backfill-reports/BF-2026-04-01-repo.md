## Current-State Summary

### Observed Product Behavior

GRCTool is a Go CLI that authenticates to Tugboat Logic through a browser-based
Safari flow, syncs policies/controls/evidence tasks into local JSON storage,
and supports evidence generation, evaluation, review, setup, and submission.
The command surface also exposes a large registry of evidence tools for
Terraform, GitHub, Google Workspace, and evidence-management workflows.

### Architecture and Runtime Shape

The current runtime is command-driven and registry-oriented rather than purely
layered in the abstract. `main.go:31-37` delegates to `cmd.Execute()`. `cmd/root.go:49-66`
boots config/logging and the global registries, while `cmd/tool.go:29-146`
builds a standardized tool execution context. Provider registration lives in
`internal/providers/registry.go:30-157`, tool registration in
`internal/tools/registry.go:27-253`, and Safari/Tugboat auth in
`internal/auth/browser.go:31-175`.

### Operational Context

The repo contract expects Go 1.24 (`go.mod:3-4`), a 4-tier test model and
`make test-no-auth` for fast feedback (`AGENTS.md`, `Makefile`), and structured
logging with secret redaction (`AGENTS.md`, `cmd/root.go:96-257`). CI and
release workflows still advertise Go 1.23 in several places
(`.github/workflows/ci.yml:9`, `.github/workflows/testing.yml`, `.github/workflows/release.yml:11`),
which is a real implementation/documentation drift.

### Evidence vs Inference

- Direct evidence: CLI bootstrap, tool context, provider registry, auth flow,
  Make targets, test directories, and CI workflow files
- Strong inference: repository is evolving toward broader provider abstractions,
  but current implementation remains Tugboat-centric
- Low-confidence area avoided during canonization: future “system of record”
  roadmap details beyond what current docs intentionally state
